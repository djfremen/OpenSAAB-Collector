# OpenSAAB-Collector v0.2.0 — USBPcap integration

**Status:** scoped, not yet implemented
**Date opened:** 2026-05-15
**Driver:** today's T8 vs ME9.6 ECM walk made it clear we need *byte-level* USB ground-truth alongside the existing PDU-level shim log. With USBPcap captured during a walk, the data anchors three independent layers (operator action / PDU calls / USB wire), which makes (a) chipsoft-envelope validation, (b) Path A `LibUsbTransport` for the operator API, and (c) "did this byte actually leave the PC" questions all answerable from a single bench session.

## Goal

Add a one-click USB capture surface to the existing tray app so a contributor can start a USBPcap capture, run a Tech2Win action (or any J2534 client), stop the capture, and have the resulting `.pcapng` upload to the same `/ingest` pipeline the shim logs already use. No second tool, no command line, no manual file shuffling.

## Scope

### In

- New tray menu items:
  - `🟢 Start USB capture` — disabled while a capture is running.
  - `🔴 Stop USB capture` — disabled while no capture is running.
  - Optional `Show captures folder…` shortcut to `%TEMP%`.
- Service-side: `UsbPcapCaptureService` (sibling of the existing file-watcher), spawns + supervises `USBPcapCMD.exe`, owns rotation/gzip/upload identically to the cstech2win path.
- Device filter: pre-narrowed to Chipsoft Pro (`VID 0x0483 / PID 0x5740`) at capture-start so the `.pcapng` only contains the device we care about. Filter expression goes through USBPcap's `--devices` argument.
- File naming: `%TEMP%\opensaab-usbpcap-<wall_ms>.pcapng`. On Stop: 30 s settle, gzip, POST to `/ingest/shim-log` with `X-Capture-Source: usbpcap`, `X-Install-ID: <install_id>`, `X-Consent-Version: <ver>`.
- Server-side (`saab-security-api`): add `usbpcap` to the accepted `X-Capture-Source` whitelist; route through R2 like cstech2win. Live decode via existing `Chipsoft_RE/tools/decode_chipsoft_pcap.py` triggered by `/summary`.
- Consent screen update: explicit line that USBPcap captures USB traffic, that the Collector pre-filters to the Chipsoft device so no other USB peripheral is recorded, and that capture is opt-in per session (same model as shim-log upload).
- Installer: chain the USBPcap MSI from desowin.org during the Collector Inno setup. Detect existing install via registry; skip if already present. Document the one-time reboot requirement on first install (USBPcap installs a kernel-mode filter driver).

### Out

- No bus-side multiplexing — only one capture (cstech2win shim log OR USBPcap) at a time on the same Chipsoft. We will revisit if running both simultaneously turns out to be reliable.
- No GUI controls beyond the tray menu items in v0.2.0. Pause/resume mid-capture isn't needed for now; Stop and Start again is fine.
- No automatic decode/render inside the tray. Decode happens server-side after upload; the tray just confirms upload success.

## Architecture

```
                  Tray app (WinForms)
                       │
              ┌────────┴────────┐
              │ start / stop    │
              │ (named pipe)    │
              ▼                 │
   OpenSAAB.Collector.Service   │
   ┌─────────────────────────┐  │
   │ FileWatcherService      │  │  (existing — cstech2win shim logs)
   │   %TEMP%\cstech2win_*   │  │
   └─────────────────────────┘  │
   ┌─────────────────────────┐  │
   │ UsbPcapCaptureService   │◄─┘  (new)
   │   spawns USBPcapCMD.exe │
   │   waits for Stop        │
   │   on Stop → rotate file │
   └─────────────┬───────────┘
                 │ rotated file
                 ▼
   ┌─────────────────────────┐
   │ Uploader (existing)     │
   │  gzip + POST /ingest    │
   │  X-Capture-Source:      │
   │   "cstech2win" OR       │
   │   "usbpcap"             │
   └─────────────────────────┘
```

The new service has the same lifecycle as the existing watcher: started by the Windows Service host on boot, takes Start/Stop commands over the existing tray↔service named pipe.

## USBPcap CLI usage (locked-down version we'll spawn)

```
USBPcapCMD.exe ^
    --devices 1 ^
    --device-filter "(idVendor=0x0483 and idProduct=0x5740)" ^
    -o "%TEMP%\opensaab-usbpcap-%WALLMS%.pcapng"
```

Notes:
- `--devices` selects which USBPcap virtual interface to capture (`\\.\USBPcap1` etc.). Tray probes interfaces at start-time using `USBPcapCMD.exe --extcap-interfaces` and picks the one the Chipsoft is enumerated under (USBPcap reports parent hub per interface).
- `--device-filter` keeps non-Chipsoft devices on the same hub out of the file. Privacy and size both benefit.
- USBPcap writes pcapng directly, no conversion needed.

## Server-side changes

Single-file change in `saab-security-api/server.py`: the `ALLOWED_SOURCES` whitelist (currently `{"cstech2win", "j2534"}`) gains `"usbpcap"`. No new endpoint, no new schema; the existing `/ingest/shim-log` handler stores by `(install_id, source, wall_ms)`.

`/summary` for a `usbpcap` file invokes `decode_chipsoft_pcap.py` (already in `Chipsoft_RE/tools/`) instead of the cstech2win JSON decoder. Output schema is the same `{ frame_count, services, can_ids, frames[] }` so the dashboard and `/operator` console don't need to know the difference.

## Open questions for v0.2.0

- Do we expose pcap captures in the public landing-page stat? Probably yes — same counter, "n captures" agnostic to source.
- Concurrent capture: shim + USBPcap on the same chipsoft hub — does USBPcap interfere with chipsoft USB latency in a way that breaks Tech2Win? Test on bench before defaulting both on.
- What's the size budget per capture? TrionicCANFlasher pcap was ~43 MB; ECM-walk pcap should be smaller because Tech2Win polls 250 ms vs flashes thousands of frames at line rate. Target: keep under 20 MB compressed.

## Acceptance test

1. Fresh Win10 VM, install v0.2.0 (chained USBPcap install).
2. Reboot.
3. Tray menu shows new Start/Stop USB capture items.
4. Open Tech2Win, navigate to ECU Information.
5. Tray → Start USB capture.
6. Walk a few ECUs in Tech2Win.
7. Tray → Stop USB capture.
8. Verify the `.pcapng.gz` lands at `https://opensaab.com/ingest/…` (or koyeb autodomain in the interim).
9. `/api/admin/captures/<install_id>` lists the file.
10. `/summary` returns frame_count > 0 and a non-empty `can_ids[]`.
